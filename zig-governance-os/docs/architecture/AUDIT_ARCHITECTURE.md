# Audit Architecture

## Purpose

Audit logging makes Zig accountable. Every material governance action must produce a tenant-scoped audit trail that can support investigations, reports, assessments, and future certifications.

## Audit Event Table

Canonical table:

```text
audit_events
```

Fields:

- `id`
- `tenant_id`
- `actor_user_id`
- `action`
- `entity_table`
- `entity_id`
- `before_state`
- `after_state`
- `reason`
- `created_at`

## Tracked Actions

Batch 21A supports:

- `create`
- `update`
- `delete`
- `approve`
- `reject`
- `assign`
- `complete`
- `generate`
- `certify`
- `review`
- `certification`

`certification` remains for compatibility with Batch 21; `certify` is the preferred action name for future repository/service operations.

## Audit Sources

Audit events can be generated by:

- repository mutations
- service workflows
- workflow approvals
- AI-generated actions
- assessment and audit reviews
- certification decisions

## Repository Audit Layer

`packages/data-access` includes:

- `AuditRepository`
- `AuditSink`
- tenant-scoped audit emission from `TenantRepository`

The repository layer currently supports audit events for:

- create
- update
- delete
- approve
- reject
- assign
- complete
- generate
- certify
- review

## Read Model

Batch 21A adds:

```text
tenant_audit_activity
```

This view joins audit events to actor email when available and uses invoker security so tenant RLS still applies.

## Security Requirements

- Audit events are tenant-scoped.
- Audit events must not be cross-tenant visible.
- System-generated actions must include actor or service identity when available.
- AI-generated actions must include source metadata when later agent workflows are implemented.

## Future Dependencies

- Batch 22 identity sessions for real actor attribution.
- Batch 23 graph relationships for traceable impact paths.
- Workflow OS for approval, rejection, assignment, and escalation audit semantics.
